# Introduction

![Status: Decided](https://img.shields.io/badge/status-Decided-green)

This artifact defines a deterministic decision process to select the parsing backend for `PandasAdapter` with readr-like, columnar, vectorized behavior while preserving `Field` semantics and existing error handling guarantees.

The decision now explicitly evaluates a **parser-centric vectorization model** where `TypeParser` remains the canonical semantic owner and adapters execute backend-specific vectorized plans derived from parser metadata.

## 1. Decision Scope

- **DEC-001**: Evaluate backend options for `apply_types` and `read_csv` conversion execution in `brasa/fieldsets/adapters/pandas_adapter.py`.
- **DEC-002**: Preserve semantic contract from `Field` and `TypeParser` definitions in `brasa/fieldsets/field.py` and `brasa/fieldsets/type_parser.py`.
- **DEC-003**: Select one primary backend strategy and one fallback strategy.
- **DEC-004**: Decide whether vectorization logic should live primarily in `PandasAdapter` or be centralized in `TypeParser` as a backend-agnostic execution contract.

## 2. Requirements & Constraints

- **REQ-001**: Primary backend must support vectorized conversion for `date`/`datetime` with optional `format`.
- **REQ-002**: Primary backend must support vectorized conversion for `numeric` with `thousands`, `decimal`, `dec`, and `sign` semantics.
- **REQ-003**: Existing `errors` modes (`raise`, `coerce`, `ignore`) must remain behaviorally compatible.
- **REQ-004**: Output contract remains pandas `DataFrame` at public API boundary.
- **REQ-005**: Semantics must continue to be defined by `Field` + `TypeParser`; backend-specific code must not silently redefine parsing rules.
- **REQ-006**: If parser-centric model is selected, `TypeParser` must expose a deterministic vectorized contract (e.g., execution plan/spec) without breaking current scalar `parse()` API.
- **CON-001**: Runtime dependencies must remain compatible with `pyproject.toml` constraints.
- **CON-002**: Decision must be evidence-based using benchmark and parity results.

## 3. Candidate Backends

| Candidate ID | Backend | Runtime Dependency Impact | Current State |
|--------------|---------|---------------------------|---------------|
| CAND-001 | Pandas-native vectorized operations | None (already required) | Evaluated |
| CAND-002 | Arrow-first (`pyarrow.csv` + Arrow compute + convert to pandas) | None for baseline dependency; integration complexity increase | Evaluated |
| CAND-003 | Polars expression engine + convert to pandas | Adds optional dependency if promoted to runtime | Skipped (polars not installed) |
| CAND-004 | Parser-centric vectorized execution (`TypeParser` defines vectorized transform spec; adapter executes backend plan) | None if pandas-backed; medium design change | **Selected** |

## 4. Evaluation Protocol

- **EVAL-001**: Use fixed-seed synthetic datasets generated by `tests/benchmarks/benchmark_field_parsers.py`.
- **EVAL-002**: Execute all candidates on identical datasets and schema sets.
- **EVAL-003**: Collect parse-time metrics for 100K, 1M, and 5M row scenarios.
- **EVAL-004**: Compare semantic parity against canonical `Field.parse` reference behavior.
- **EVAL-005**: Record memory peak and failure-mode parity.
- **EVAL-006**: Measure adapter complexity reduction (converter callbacks removed, number of adapter type-specific branches removed).
- **EVAL-007**: Verify extensibility: registering a custom parser via `TypeParserFactory.register_parser` can optionally provide vectorized execution without adapter edits.

## 8. Decision

- **FINAL-001**: Primary backend selection: **CAND-004 (Parser-centric vectorized execution)**.
- **FINAL-002**: Fallback backend selection: **CAND-001 (Pandas-native vectorized)** as scalar fallback for custom/unknown types.
- **FINAL-003**: Decision confidence level: **High** — both GATE-001 (≥3x speedup) and GATE-002 (100% parity) achieved; ARG-006 priority ordering applied.
- **FINAL-004**: Required follow-up actions:
  - Mark all Phase 0–4 tasks in `refactor-pandas-adapter-1.md` as completed.

## 10. Risks & Mitigations

- **RISK-001**: Backend semantic drift on locale-like numeric parsing.
  - **MIT-001**: Keep canonical parity tests against `Field.parse` outputs.
- **RISK-002**: Divergent null handling (`pd.NA`, `NaN`, `NaT`) across backends.
  - **MIT-002**: Add explicit null normalization stage before dtype casting.
- **RISK-003**: Increased maintenance burden from multi-backend support.
  - **MIT-003**: Select one primary backend and a minimal fallback strategy.
- **RISK-004**: Pushing execution logic into parser layer may blur responsibilities and over-couple parser abstractions to pandas internals.
  - **MIT-004**: Keep parser-facing contract backend-agnostic (plan/spec), execute concrete operations in adapter backend modules.
- **RISK-005**: Custom registered parsers might not provide vectorized plans initially.
  - **MIT-005**: Require safe scalar fallback path and explicit capability checks.

## 12. Architectural Argument: Parser-Centric Vectorization

- **ARG-001 (Why this is attractive)**: `Field` already delegates conversion semantics to `_parser`; vectorization in parser contract keeps one semantic source of truth and reduces duplicated parsing rules in `PandasAdapter`.
- **ARG-002 (Expected effect)**: `PandasAdapter` shifts from "parse executor" to "execution orchestrator", reducing row-wise callbacks (`Series.apply`) and type-specific branching.
- **ARG-003 (Extensibility)**: New parser types registered in `TypeParserFactory` can participate in vectorized execution through parser-provided plans, reducing adapter churn.
- **ARG-004 (Main caveat)**: Directly embedding pandas operations in `TypeParser` would couple parser abstractions to a backend.
- **ARG-005 (Recommended compromise)**: Keep scalar `parse()` untouched; add optional parser-level vectorized plan metadata that backend adapters compile/execute.
- **ARG-006 (Decision default)**: Prefer CAND-004 as **primary strategy** only if gates `GATE-001..GATE-006` pass; otherwise fallback to CAND-001 with strict parity tests.
